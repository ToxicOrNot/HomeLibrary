# Домашняя библиотека

Веб-приложение для учета домашней библиотеки.

## Возможности

- добавление книг в личную библиотеку;
- добавление книг в список желаемого;
- указание цикла книги, если книга входит в серию;
- поиск по названию, автору, году и заметке;
- стандартная группировка по автору, затем по циклу, затем по книгам;
- алфавитная сортировка авторов и циклов;
- порядок книг внутри цикла по добавлению;
- ручное изменение порядка книг внутри одного цикла;
- группировка списков по автору или циклу;
- выпадающие подсказки существующих авторов и циклов;
- фильтрация циклов по выбранному автору;
- оценка книг от 1 до 10;
- редактирование уже добавленных книг;
- удаление книг из любого списка;
- перенос книги из желаемого в библиотеку;
- автоматическое сохранение данных в `data/library_data.json`;
- бэкап файла данных в GitHub после каждого изменения при настроенном токене.

## Запуск

Откройте `run_library.bat` двойным кликом.

После запуска в окне появятся два адреса:

- `http://127.0.0.1:8000` — открыть на этом компьютере;
- `http://ВАШ-IP:8000` — открыть на телефоне.

Телефон и компьютер должны быть подключены к одной Wi-Fi сети. Окно сервера нужно держать открытым, пока вы пользуетесь приложением.

Если Windows спросит разрешение для сети, разрешите доступ для Python в частной сети.

## Запуск из PowerShell

```powershell
py web_library.py
```

## Размещение на хостинге

Приложение готово к запуску на Python-хостингах без внешних зависимостей.

### Streamlit Cloud

Для Streamlit Cloud запускайте файл:

```text
streamlit_app.py
```

В настройках Streamlit Cloud в поле `Main file path` укажите:

```text
streamlit_app.py
```

Не указывайте `web_library.py` для Streamlit Cloud: этот файл сам поднимает HTTP-сервер и подходит для Render/Railway/VPS, но Streamlit Cloud запускает свой сервер.

GitHub-бэкап на Streamlit Cloud настраивается в `Manage app` → `Settings` → `Secrets`:

```toml
GITHUB_TOKEN = "ваш_новый_токен"
GITHUB_REPOSITORY = "owner/repo"
GITHUB_BRANCH = "main"
GITHUB_BACKUP_PATH = "data/library_data.json"
```

После добавления книги приложение показывает статус последнего GitHub-бэкапа над списком книг.

### Render

1. Загрузите проект в GitHub.
2. Создайте на Render новый `Web Service`.
3. Выберите репозиторий с проектом.
4. Укажите:
   - Runtime: `Python 3`
   - Build command: `pip install -r requirements.txt`
   - Start command: `python web_library.py`
5. После деплоя откройте выданный Render URL.

### Railway

1. Загрузите проект в GitHub.
2. Создайте Railway project из репозитория.
3. Railway сам использует переменную `PORT`.
4. Start command: `python web_library.py`.

### Данные и GitHub-бэкап

Книги хранятся отдельно от кода:

```text
data/library_data.json
```

При первом запуске приложение перенесет старый `library_data.json` в `data/library_data.json`, если новый файл еще не создан.

Приложение делает бэкап этого файла в GitHub после каждого изменения списка книг или желаемого. При запуске приложения также есть дополнительная проверка автоматического бэкапа, но не чаще одного раза в 24 часа.

Для бэкапа задайте на хостинге переменные окружения:

- `GITHUB_TOKEN` — GitHub fine-grained token с правом `Contents: Read and write` для этого репозитория;
- `GITHUB_REPOSITORY` — репозиторий в формате `owner/repo`;
- `GITHUB_BRANCH` — ветка, обычно `main`;
- `GITHUB_BACKUP_PATH` — путь файла в репозитории, обычно `data/library_data.json`.

Опционально:

- `DATA_FILE` — путь к файлу данных, если хостинг дает persistent disk;
- `GITHUB_BACKUP_INTERVAL_SECONDS` — интервал дополнительного бэкапа при запуске, по умолчанию `86400`.

Если `GITHUB_TOKEN` или `GITHUB_REPOSITORY` не заданы, приложение просто сохраняет данные локально и не пытается делать GitHub-бэкап.
